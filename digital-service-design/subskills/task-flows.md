# Task Flows Subskill

## Purpose

Convert a service into a clear sequence of resident actions. Use this subskill to remove ambiguity, reduce unnecessary steps, and make task completion easier.

## When to use

Use when creating or reviewing step-by-step instructions, service pages, application flows, payment flows, appointment flows, reporting workflows, or support procedures.

## Core rules

- Write steps in the order residents perform them.
- Start each step with a verb.
- Keep each step to one action when possible.
- Separate required steps from optional, conditional, or follow-up steps.
- State what the resident needs before each step.
- State what the resident receives after each step.
- Avoid presenting County back-office steps as resident steps unless the resident must act on them.

## Flow types

### Linear flow

Use when every resident follows the same path.

Example structure:

1. Check whether you are eligible.
2. Gather required documents.
3. Complete the application.
4. Pay the fee.
5. Submit your application.
6. Save your confirmation number.
7. Wait for review.

### Branching flow

Use when the next step depends on eligibility, location, service type, fee status, prior approval, account status, or documentation.

Format branches as:

- If [condition], [action].
- If [condition], [different action].
- If you are not sure, [help path].

### Channel-choice flow

Use when residents can complete a task online, by phone, by mail, by email, or in person.

For each channel, state:

- Who should use it
- What they need
- How long it takes
- Whether the result is the same
- What confirmation they receive

## Task-flow review questions

- Can the resident tell what to do first?
- Are steps ordered by resident action rather than department process?
- Are there hidden prerequisites?
- Are required documents introduced before the form begins?
- Is payment placed at the right point in the flow?
- Does the flow explain what happens after submission?
- Are exceptions, alternatives, and recovery paths visible?
- Are there duplicate steps or repeated data entry?
- Is every step necessary?

## Output: task-flow outline

Use this structure:

## Task
[Resident action, such as “Apply for...”]

## Before you start
- Eligibility:
- Documents:
- Fees:
- Deadline:
- Estimated time:

## Steps
1. [Action]
   - What you need:
   - What happens after this step:
2. [Action]
   - What you need:
   - What happens after this step:

## If this does not work
- If [problem], [recovery action].
- If [problem], [recovery action].

## After you finish
- Confirmation:
- Processing time:
- Status check:
- Contact:

## Quality criteria

A task flow is ready when residents can complete the task without needing to infer missing prerequisites, internal process, timing, or recovery options.
