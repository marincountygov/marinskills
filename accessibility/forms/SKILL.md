# Form Accessibility Skill

Use this skill to create, review, or remediate accessible County of Marin forms, including online forms, fillable PDFs, and document-based forms.

## Purpose

Forms are often the point where residents request services, apply for benefits, register, pay, report, appeal, or provide required information. They must be accessible, understandable, and easy to complete.

## When to use this skill

Use this skill for:

- Web forms.
- Fillable PDF forms.
- Word or Google Doc forms.
- Application forms.
- Registration forms.
- Complaint, request, report, and appeal forms.
- Payment or checkout flows.
- Eligibility screeners.
- Multistep transactions.

## Accessibility requirements

Forms should:

- Have clear labels for every field.
- Programmatically associate labels with fields.
- Use fieldsets and legends for grouped questions.
- Provide clear instructions before they are needed.
- Identify required fields accessibly.
- Provide helpful error messages.
- Move focus logically and predictably.
- Be fully keyboard operable.
- Avoid time limits unless necessary and accessible.
- Preserve entered information after validation errors.
- Use autocomplete where appropriate.
- Avoid placeholder-only labels.
- Use sufficient color contrast.
- Avoid relying on color alone for errors or status.
- Provide confirmation and next steps after submission.

## Plain-language requirements

Forms should:

- Ask only for information that is necessary.
- Use familiar words for field labels.
- Explain why sensitive information is requested.
- Define eligibility and documentation requirements before the user begins.
- Break long forms into logical sections.
- Use direct calls to action.
- Provide estimated completion time when useful.

## Form design requirements

Forms should:

- Follow a logical sequence.
- Group related fields.
- Use appropriate input types.
- Avoid unnecessary optional fields.
- Make help text concise and specific.
- Clearly distinguish save, submit, cancel, and reset actions.
- Avoid reset buttons unless there is a strong reason.

## Review workflow

1. Identify the service goal and user task.
2. Review the question set for necessity and clarity.
3. Check labels, groups, instructions, and required fields.
4. Test keyboard operation and focus order.
5. Review validation and error recovery.
6. Confirm confirmation message and next steps.
7. Check plain language, brand, and privacy-sensitive content.

## Output format

When reviewing a form, provide:

- Accessibility status.
- Task-completion risks.
- Field-level issues.
- Error-message issues.
- Plain-language revisions.
- Privacy or governance flags.
- Recommended remediation steps.
