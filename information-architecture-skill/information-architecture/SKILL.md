# Information Architecture Skill

## Purpose

Improve findability, navigation, page placement, internal linking, duplication control, and content taxonomy.

Use this skill when asked to assess, create, revise, or govern website structures, navigation systems, page homes, link relationships, search result snippets, canonical pages, or topic taxonomies.

## When to use this skill

Use this skill for requests involving:

- Where a page or service should live on a website
- Navigation labels, menu labels, breadcrumbs, or section names
- Cross-linking between related pages, services, departments, forms, events, news, and contact points
- Search result titles, summaries, and snippets
- Duplicate, overlapping, outdated, or competing pages
- Canonical page selection and consolidation plans
- Topic taxonomy, tags, facets, categories, filters, and controlled vocabulary
- Findability reviews for residents, businesses, employees, partners, and other public audiences

Do not use this skill as a substitute for content strategy, plain language, accessibility, visual design, or digital service design. Route those concerns to the relevant skill when the primary issue is content purpose, readability, compliance, appearance, or transaction completion.

## Core principles

Information architecture decisions must help users answer four questions quickly:

1. Am I in the right place?
2. Is this the page or path I need?
3. What related information should I consider next?
4. What should I do if this page is not the right one?

Prioritize user mental models over internal organizational structure. Department ownership matters for governance, but user-facing structure should reflect how people look for services, tasks, topics, and life events.

## Required workflow

For any IA task, identify:

1. **User intent**: what users are trying to find or complete.
2. **Content object**: the thing being placed, labeled, linked, deduplicated, or classified.
3. **Current home**: where it currently lives, if known.
4. **Recommended home**: where it should live and why.
5. **Navigation path**: the likely route from homepage, section landing page, search, or related page.
6. **Relationship model**: parent page, sibling pages, child pages, related pages, canonical page, and duplicates.
7. **Metadata model**: topic, audience, service type, department, location, lifecycle status, and other relevant tags.
8. **Success criteria**: how findability or comprehension will be measured.

If evidence is missing, make a best-effort recommendation and state the assumptions that shaped it.

## Recommended outputs

Depending on the request, produce one or more of these outputs:

- Recommended navigation label with rationale
- Page placement recommendation
- Breadcrumb path
- Internal linking plan
- Search title and snippet
- Duplicate-content consolidation map
- Canonical page recommendation
- Topic taxonomy or tagging scheme
- IA review checklist
- Before-and-after structure
- Redirect or retirement recommendation
- Measurement plan for findability

## Decision rules

### Navigation labels

Use short, concrete labels based on user language. Avoid department jargon, program names, acronyms, slogans, and labels that require insider knowledge.

### Page placement

Place content where the primary audience expects to find it, not necessarily where the owning department sits in the organization chart.

### Cross-linking

Links should reflect meaningful user journeys. Link to pages that help users decide, act, recover, compare, or continue.

### Search snippets

Search snippets should tell users what the page is, who it is for, and what they can do there.

### Duplication and canonical pages

Do not let multiple pages compete to answer the same user question. Select one canonical page, consolidate overlapping content, and redirect or clearly demote duplicates.

### Topic taxonomy

Use taxonomy terms that are stable, mutually understandable, and useful for retrieval, filtering, routing, reporting, and governance.

## Quality checklist

Before finalizing an IA recommendation, verify that:

- The recommended structure matches likely user intent.
- Labels are understandable without organizational context.
- The primary page has one clear canonical location.
- Related links are useful, not merely exhaustive.
- Search snippets distinguish similar pages from each other.
- Taxonomy terms are consistent and not duplicative.
- Deprecated, duplicate, or outdated content has a disposition.
- The recommendation includes a way to measure whether findability improved.
