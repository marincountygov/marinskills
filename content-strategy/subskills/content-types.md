# Subskill: Content Types

## Purpose

Use this subskill to classify content by function so each item uses the right structure, lifecycle, metadata, and success measures.

## Core rule

Choose the content type based on what the user needs to do, not on what the organization wants to publish.

## Common content types

### Service page

Use when users need to complete a government or organizational task.

Typical purposes:

- Apply.
- Request.
- Pay.
- Register.
- Report.
- Renew.
- Schedule.
- Find eligibility, requirements, fees, deadlines, or next steps.

Strategy notes:

- Prioritize task completion.
- Surface requirements, cost, timing, and contact paths.
- Link to forms, portals, or service channels.
- Maintain as evergreen content unless the service changes.

Success measures:

- Completed applications, payments, registrations, or requests.
- Reduced support contacts for basic questions.
- Lower abandonment on service pathways.
- Fewer duplicate or misdirected requests.

### Guide or explainer

Use when users need to understand a process, rule, or decision before acting.

Typical purposes:

- Explain how something works.
- Compare options.
- Clarify steps.
- Prepare users for a process.

Strategy notes:

- Keep guidance task-oriented.
- Link to the service page where action happens.
- Avoid duplicating policy source text unless necessary.

Success measures:

- Reduced confusion.
- Improved progression to the correct next step.
- Lower support burden.

### Policy or official requirement

Use when the content itself is authoritative and users need the official rule.

Typical purposes:

- State requirements.
- Publish adopted policy.
- Document standards, procedures, or official determinations.

Strategy notes:

- Identify source of authority.
- Include version, effective date, owner, and review cycle.
- Provide plain-language summary separately when needed.

Success measures:

- Users can identify current applicable requirements.
- Reduced use of outdated versions.
- Fewer conflicting references.

### News or announcement

Use when timeliness is the main value.

Typical purposes:

- Announce a change.
- Notify users of a development.
- Share a recent decision or milestone.

Strategy notes:

- Do not use news as the permanent source of truth for a service, program, or policy.
- Link to evergreen content that should be updated separately.
- Define when the item becomes stale.

Success measures:

- Reach during the relevant time window.
- Click-through to permanent source or action.
- Reduced confusion during transition periods.

### Alert or urgent notice

Use when users need immediate, time-sensitive information.

Typical purposes:

- Service disruption.
- Emergency update.
- Closure.
- Deadline change.
- Safety notice.

Strategy notes:

- Use clear start and end times.
- State who is affected.
- State what users should do now.
- Define automatic removal or review trigger.

Success measures:

- Timely visibility.
- Reduced calls or misdirected visits.
- Clear user action during disruption.

### Event

Use when users need to attend, register for, or know about a scheduled occurrence.

Typical purposes:

- Meeting.
- Workshop.
- Public hearing.
- Training.
- Community event.

Strategy notes:

- Include date, time, location, registration, accessibility accommodations contact if handled by a separate standard, agenda, and follow-up materials when applicable.
- Archive or convert after the event.

Success measures:

- Registrations or attendance.
- Agenda/document access.
- Post-event follow-up completion.

### Campaign or landing page

Use when users need a temporary or thematic entry point across multiple pieces of content.

Typical purposes:

- Seasonal program.
- Public awareness campaign.
- Initiative launch.
- Multi-channel communications push.

Strategy notes:

- Define campaign window.
- Link to permanent source pages.
- Do not duplicate large amounts of evergreen content.
- Define unpublish, archive, or conversion plan before launch.

Success measures:

- Campaign conversions.
- Click-through to key actions.
- Reach within target audience.
- Completion of campaign-specific outcomes.

### Form, document, or downloadable asset

Use when the file or artifact is necessary for a transaction, record, or offline workflow.

Strategy notes:

- Avoid using downloadable files as substitutes for web pages when the content is meant to be read online.
- Assign source of truth.
- Define version control and replacement process.
- Link from relevant service or guide pages.

Success measures:

- Correct version downloads.
- Reduced outdated-file use.
- Fewer incomplete or incorrect submissions.

### Directory or index

Use when users need to find people, services, locations, documents, or related pages.

Strategy notes:

- Define inclusion criteria.
- Use consistent metadata.
- Provide filters or groupings when the set is large.
- Remove stale entries.

Success measures:

- Search refinement success.
- Click-through to correct item.
- Reduced failed searches.

## Content type selection questions

Ask:

- What user action or decision does this support?
- Is this content evergreen, seasonal, urgent, scheduled, or archival?
- Is this the source of truth or a pointer to the source of truth?
- Does this content need a review date, expiration date, or event date?
- Does the content need structured metadata or taxonomy?
- Is a page better than a document?
- Should this be merged with existing content?
