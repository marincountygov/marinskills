# Subskill: Page Purpose

## Purpose

Use this subskill to define why a page or content asset exists and what it must accomplish for users and the organization.

## Core rule

A page should have one primary purpose. If a page has several unrelated purposes, recommend splitting, restructuring, or creating clearer pathways.

## Page purpose statement

Use this format:

> This page exists to help [audience] [complete task or answer question] so they can [user outcome], while supporting [organizational purpose].

Example:

> This page exists to help property owners understand whether they need a permit and start the correct application so they can avoid delays, while supporting accurate permit intake.

## Purpose components

### Audience

Who the page primarily serves.

### User task or question

What the user needs to do or understand.

### User outcome

What becomes easier, clearer, safer, faster, or more accurate for the user.

### Organizational purpose

What valid organizational need the content supports, such as service delivery, compliance, participation, transparency, intake quality, or reduced support burden.

## Good page purposes

Strong page purposes are:

- Specific.
- User-centered.
- Actionable.
- Measurable.
- Maintained by an owner.
- Distinct from nearby pages.

## Weak page purposes

Revise purposes that only say:

- “Provide information.”
- “Promote awareness.”
- “Tell people about our program.”
- “House resources.”
- “Explain what we do.”

These may be starting points, but they are not specific enough for strategy.

## Page purpose types

### Transactional

The page helps users complete a task.

Indicators:

- Apply, pay, renew, report, request, register, submit.

Required strategy elements:

- Eligibility.
- Requirements.
- Steps.
- Fees.
- Deadlines.
- Processing time.
- Confirmation or next step.

### Informational

The page helps users understand a topic or decision.

Required strategy elements:

- Clear answer to the primary question.
- Scope and limits.
- Related actions.
- Source of truth.

### Navigational

The page helps users choose where to go next.

Required strategy elements:

- Distinct pathways.
- Descriptive link labels.
- Metadata or grouping logic.
- Criteria for choosing a path.

### Participatory

The page helps users engage in a process.

Required strategy elements:

- Who can participate.
- How to participate.
- Dates and deadlines.
- What input affects.
- Where outcomes are published.

### Accountability or transparency

The page documents decisions, performance, records, budgets, reports, or official information.

Required strategy elements:

- Authority or source.
- Date/version.
- Owner.
- Related records.
- Update cadence.

## Page purpose review questions

Ask:

- What user problem does this page solve?
- What would users fail to do if this page did not exist?
- What is the one most important thing users should be able to do after using it?
- Is the page the source of truth or a route to the source of truth?
- Does this purpose overlap with another page?
- What should happen to the page when the purpose expires?

## Decision rules

- If no user task or question can be stated, do not recommend publishing the page as proposed.
- If the page exists mainly for internal visibility, consider whether it belongs on an internal channel.
- If the page duplicates another page, recommend merge, redirect, or clarify distinct purposes.
- If the page is time-limited, define a sunset or archive trigger.
